### 3. SOLID-принципы ООП
---
##### **SOLID — аббревиатура пяти принципов проектирования в ООП.**

---
##### **S — Single Responsibility (принцип единственной ответственности)**  
**У каждого класса должна быть только одна зона ответственности**. Всё поведение объекта направлено исключительно на её обеспечение, что позволяет избежать непредвиденных изменений в системе.
Например:
- класс `User` хранит данные пользователя;
- класс `UserDatabase` отвечает только за сохранение пользователя в БД.
---
##### **O — Open-Closed (принцип открытости/закрытости)**  
Классы должны быть **открыты для расширения**, но **закрыты для изменения**. Внесение изменений в существующий компонент может вызвать непредсказуемые последствия для системы, поэтому лучше наследоваться и расширять функциональность, а не переписывать исходный код.

---
##### **L — Liskov Substitution (принцип подстановки Лисков)**  
Объекты дочернего класса должны полностью заменять объекты родительского без изменения поведения программы.
Например:
```
class Animal:  
	def speak(self):  
		...
		
class Dog(Animal):  
	def speak(self):  
		print("Гав")
```

Любая функция, работающая с `Animal`, должна корректно работать и с `Dog`.

---
##### **I — Interface Segregation (принцип разделения интерфейса)**  
Лучше сделать несколько небольших специализированных интерфейсов, чем один большой универсальный.
Клиент должен зависеть только от тех методов, которые действительно использует.

---
##### **D — Dependency Inversion (принцип инверсии зависимостей)**  
Высокоуровневые модули должны зависеть **не от конкретных классов**, а от **абстракций**.
Это уменьшает связанность компонентов программы и облегчает замену реализаций.
